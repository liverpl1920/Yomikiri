# Yomikiri
## サービス概要
* 技術書やビジネス書の「積読（つんどく）」に特化し、本に**賞味期限（読了期限）**を設定する管理アプリです。
* Googleカレンダーと連携して、「読書タスク」を予約し、物理的な読書時間を確保します。
  * MVP：ワンクリックで予定作成画面へ遷移
  * 本リリース：ユーザー指定の時間帯に自動挿入
  * 将来：空き時間を解析して最適な時間に自動挿入
* 期限が迫ると通知やUIの変化で適度なプレッシャーを与え、**「いつか読む」を「今読む」に変える**実行支援を提供します。

## このサービスへの思い・作りたい理由
**【背景】**
エンジニアを目指して学習を始めてから、良書と呼ばれる技術書を次々と購入しましたが、気づけばデスクの脇には手付かずの本が山積みになっていました。一冊読み終える前に新しい本に目移りしてしまい、結局どれも中途半端になる自分に強い罪悪感を感じていました。

**【作りたい理由】**
「読みたい気持ちはあるのに、時間が作れない」という悩みは、単なる意志の弱さではなく、**「読書を具体的なスケジュールに落とし込めていないこと」**が原因だと考えました。既存の管理アプリは「記録」には適していても、「読ませる」ための強制力が足りません。あえて「期限」という制約をシステム側で設けることで、過去の自分と同じように積読に悩む人の背中を押し、確実なスキルアップを支援したいという思いから本サービスを考案しました。

## ユーザー層について
1. **プログラミング初学者・現役エンジニア**
    * 理由：技術の進歩が速く、インプットすべき情報（本）が膨大であるため、最も積読の悩みと「情報の賞味期限」の概念が切実な層だからです。
2. **資格試験や自己研鑽に励むビジネスパーソン**
    * 理由：「いつか読もう」ではなく「試験日までに読まなければならない」という明確な目標を持つ層にとって、期限管理機能が強い価値を持つからです。
3. **「本を買って満足してしまう」自覚がある読書家**
    * 理由：収集自体が目的化している層に対し、未読本を「負債」として可視化することで、行動変容を促すきっかけを提供できるからです。

## サービスの利用イメージ
#### MVP：
1. 本を登録して期限設定
2. 「Googleカレンダーに追加」を押す → 予定作成画面へ遷移（日時はユーザーが決定）
3. 毎日必要ページ数を見ながら進捗更新
4. 読了にして一覧がスッキリ（達成感）

#### 本リリース：
1. 買ったばかりの技術書を登録し「2週間後」を賞味期限に設定
2. 読書可能な時間帯（例：平日19:00-21:00）を登録
3. Googleカレンダー連携 → 指定時間帯に30分の読書枠が自動挿入される
4. 期限3日前、メールで「あと3日です」と通知。書影もセピア調に変化
5. 積読タイムラインで他のユーザーの進捗を見てモチベーションアップ
6. 読了時にシェア用ページを生成（OGP対応）

## ユーザーの獲得について
* **SNS（X / Twitter）での発信：** 「#エンジニアの積読」などのハッシュタグを活用し、読了記念のシェア機能で似た悩みを持つ層に認知を広げます。
* **技術コミュニティへの投稿：** QiitaやZennなどの媒体で「なぜ不便な積読アプリを作ったのか」という技術記事を執筆し、開発者層の共感と利用を誘います。

## サービスの差別化ポイント・推しポイント

### 【差別化ポイント】
#### 1) 「記録」ではなく「予定化」まで面倒を見る
既存の読書管理サービス（ブクログ等）は、読んだ本・読みたい本の**ログ**が中心で、「読む行動」はユーザー任せになりがちです。  
Yomikiriは、登録した本を**読書タスクとして予定化**し、「読む時間」を先に確保するところまで支援します。

- **MVP：** ワンクリックで「Googleカレンダーの予定作成画面（テンプレ入力済み）」へ遷移  
- **本リリース：** ユーザーが登録した読書可能時間帯に、30分枠の読書予定を自動挿入  
- **将来：** freebusy APIで空き時間を解析し、最適な時間に自動挿入  

#### 2) 「期限」を“行動に変換できる形”で提示する
ただ締切日を見せるだけでなく、期限から逆算して**今日やるべき量**まで落とし込みます。

- 残ページ数・残日数から、**1日あたりの必要読書ページ数（ノルマ）**を自動計算  
- 進捗更新に応じてノルマも更新されるため、「何をどれだけ読めばいいか」が常に明確になります

#### 3) 期限の緊急度を“体感できる”UI（賞味期限ビジュアライザー）
期限が近づくほど書影が色褪せる（セピア＋透明度）ことで、**一覧を見ただけで「早く読まないとまずい本」**が分かります。  
数字の締切を“体感”に変えて、先延ばしを防ぎます。

#### 4) 相互の刺激で継続を支援する（積読タイムライン）
本リリースでは、共有設定した本のみがタイムラインに表示され、他のユーザーの進捗や「期限への焦り」が可視化されます。  
一人では挫折しがちな読書を、**相互の刺激**によって習慣化させます。

### 【推しポイント】
#### 推し1：意志に頼らず、予定化で「読む時間」を先に押さえる
「いつ読もうか」を考える認知負荷が減り、忙しくても後回しになりにくくなります。  
MVPでも**予定作成までの手間を最小化**し（テンプレ入力済み）、本リリースでは自動挿入でさらに強制力を高めます。

#### 推し2：「今日は何ページ？」が出るから、着手が軽い
期限と進捗からノルマが出ることで、「時間がないから無理」を「今日は◯ページ読めばOK」に変換できます。  
迷いが減り、日々の読書が“タスク化”されます。

#### 推し3：期限が近いほど焦る設計で、先延ばしを抑える
賞味期限ビジュアライザーにより、期限が迫るほど視覚的に緊急度が上がります。  
締切を「ただの数字」にしないことで、行動を引き出します。

## 機能候補
### MVPリリース時
* **会員登録・ログイン機能：** Rails標準の認証（Devise等）によるアカウント管理。
* **積読一覧表示：** 現在登録されている積読本を、期限が近い順にソートして表示。
* **積読登録：** 書籍情報（タイトル・画像・総ページ数）と「読了期限」の登録。
* **積読詳細：** 特定の本の進捗状況、1日あたりの必要読書ページ数の確認。
* **読書ステータス管理：** 未読・読書中・読了の切り替え機能。
* **Googleカレンダー連携（簡易版）：** 読書予定ごとに「Googleカレンダーに追加」リンクを生成。ワンクリックでカレンダーのイベント作成画面が開く。
* **「賞味期限」ビジュアライザー（CSS版）：** 期限までの日数に応じて書影が色褪せるCSSエフェクト（filter: sepia + opacity調整）。

### 本リリース時
* **Googleカレンダー連携（自動版）：** OAuth認証後、ユーザーが指定した読書可能時間帯に読書スロットを自動挿入。
* **ISBN検索：** openBD APIによる書籍情報の自動取得・補完（APIキー不要で利用可能）。
* **メール通知機能：** ActionMailerを用いた、賞味期限前のリマインド通知。
* **読了シェア機能（OGP対応）：** 読了時にOGPメタタグ付きのHTMLページを生成。SNSシェア時にタイトル・読了日数が表示される。
* **積読タイムライン機能：** 他のユーザーの積読状況や進捗を閲覧可能。共有設定した本のみ表示。
* **読書スピード計測：** タイマー機能で1ページあたりの読書時間を計測・保存し、ノルマ計算の精度を向上。
* **バックグラウンド処理：** GitHub Actions（cron）でrake叩くことで定期実行を実現（毎日の期限チェック・通知送信）。
  * 失敗時はGitHub Actionsの通知で検知し、再実行できる運用を想定

### 将来的な拡張（ユーザー要望に応じて）
* **Googleカレンダー連携（空き時間解析版）：** freebusy APIで空き時間を解析し、最適な時間に自動挿入。
* **LINE通知機能：** LINE Messaging APIを用いたプッシュ通知。
* **OGP画像生成：** Cloudinaryの動的変換機能で読了実績を画像化し、SNSでの視認性を向上。
* **「私の積読山」シェア：** 積読合計金額や期限切れ間近の本をOGP画像化してSNS投稿。
* **積読タイムラインのリアクション機能：** 他のユーザーの積読にコメントやスタンプでリアクション。
* **読書イベント機能：** 期間限定の共有ルームで、同じ本を読むユーザー同士が進捗を共有。
* **ジョブキュー導入：** 即時非同期処理やリトライが必要になった場合、solid_queueへ移行。

## 使用する技術スタック
### 実装イメージ（CRUD以外）
* **Googleカレンダー連携：**
  * MVP（簡易版）：Google Calendar の event create URL を生成して遷移（OAuth不要）
  * 本リリース（自動版）：OmniAuth + google-api-client でOAuth認証し、ユーザー指定時間帯に events.insert
  * 将来（空き時間解析版）：freebusy API で空き時間を取得し、最適な時間に自動挿入
* **ノルマ計算ロジック：** 残りページ数を $P$、残り日数を $D$ とした時、1日あたりの必要ページ数 $N$ を以下の式で算出し、毎日バッチ処理で更新します。
$$N = \lceil P / D \rceil$$
* **バックグラウンド処理：** GitHub Actions（cron）でrake叩き、毎日決まった時間に期限チェック・通知送信を実行。
