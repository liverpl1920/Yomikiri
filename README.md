# Yomikiri

## サービス概要

技術書やビジネス書の「積読（つんどく）」に特化し、本に**賞味期限（読了期限）**を設定する管理アプリです。

### 主な特徴

| 特徴 | 説明 |
|------|------|
| 賞味期限の設定 | 本に読了期限を設定し、期限意識を持たせる |
| Googleカレンダー連携 | 読書タスクを予約し、読書時間を"予定として見える化" |
| 賞味期限ビジュアライザー | 期限が迫るとUIが変化し、適度なプレッシャーを与える |

#### Googleカレンダー連携の進化

| フェーズ | 内容 |
|----------|------|
| MVP | ワンクリックで予定作成画面へ遷移（アプリ内にカレンダーUIは持たない） |
| 本リリース | ユーザー指定の時間帯に自動挿入 |
| 将来 | 空き時間を解析して最適な時間に自動挿入 |

#### 通知機能の進化

| フェーズ | 内容 |
|----------|------|
| MVP | UI変化（賞味期限ビジュアライザー）を中心に提供 |
| 本リリース | メール通知（リマインド）を追加 |

> **Note**  
> 本サービスはアプリ内にカレンダーUIを持たず、普段使っているGoogleカレンダーに読書予定を追加することで、読書時間を確保します。

## このサービスへの思い・作りたい理由

### 背景

エンジニアを目指して学習を始めてから、良書と呼ばれる技術書を次々と購入しましたが、気づけばデスクの脇には手付かずの本が山積みになっていました。一冊読み終える前に新しい本に目移りしてしまい、結局どれも中途半端になる自分に強い罪悪感を感じていました。

### 作りたい理由

「読みたい気持ちはあるのに、時間が作れない」という悩みは、単なる意志の弱さではなく、**「読書を具体的なスケジュールに落とし込めていないこと」**が原因だと考えました。

既存の管理アプリは「記録」には適していても、「読ませる」ための強制力が足りません。あえて「期限」という制約をシステム側で設けることで、過去の自分と同じように積読に悩む人の背中を押し、確実なスキルアップを支援したいという思いから本サービスを考案しました。

## ユーザー層について

| ユーザー層 | 理由 |
|------------|------|
| プログラミング初学者・現役エンジニア | 技術の進歩が速く、インプットすべき情報（本）が膨大であるため、最も積読の悩みと「情報の賞味期限」の概念が切実な層 |
| 資格試験や自己研鑽に励むビジネスパーソン | 「いつか読もう」ではなく「試験日までに読まなければならない」という明確な目標を持つ層にとって、期限管理機能が強い価値を持つ |
| 「本を買って満足してしまう」自覚がある読書家 | 収集自体が目的化している層に対し、未読本を「負債」として可視化することで、行動変容を促すきっかけを提供できる |

## サービスの利用イメージ

### MVP

#### 1. 本を登録して期限設定

| 項目 | 必須 | 説明 |
|------|------|------|
| タイトル | ○ | 本のタイトル |
| 著者 | - | 著者名（未入力可） |
| 総ページ数 | ○ | 本の総ページ数 |
| 読了対象ページ数 | ○ | 初期値＝総ページ数。索引や付録を除いた範囲を設定可能 |
| 現在ページ | ○ | 初期値0 |
| 読了期限（賞味期限） | ○ | いつまでに読み終えるか |
| 書影 | - | 未設定時はデフォルトのプレースホルダー画像を表示 |

#### 2. Googleカレンダーに追加

「Googleカレンダーに追加」ボタンを押すと、予定作成画面へ遷移します。

- 日時はユーザーが決定
- テンプレ入力（予定作成画面に事前入力される内容）：**予定タイトル / 説明文 / 所要時間（終了時刻）**
- 所要時間の初期値は30分。**15 / 30 / 45 / 60分** から選択可能
- "都度追加"方式（繰り返し予定はユーザーがGoogle側で設定可能）
- **MVPではGoogle側の予定変更/削除は追跡しない**（予定化が行動変容に効くかを先に検証）

#### 3. 進捗更新

基本は「今日読んだページ数」を入力すると、現在ページに自動反映されます（±ボタンで増減も可能）。  
（上級者向けに、現在ページを直接入力することも可能です。）

#### 4. ノルマ確認

画面上の「今日のノルマ（必要ページ数）」を確認しながら読書を継続します。

- **ノルマ計算：**  
  - 残ページ数 $P$ = 読了対象ページ数 - 現在ページ  
  - 残日数 $D$ = **今日を含む残日数（期限日を含む）**（※期限当日は $D=1$ として扱う）  
  - 1日あたりの必要ページ数 $N$ は **切り上げ** で算出  
  - $$N = \lceil P / D \rceil$$
- MVPではノルマは「ページ」で提示（時間は読書予定の枠としてのみ使用）
- MVPは毎日計算の単純モデル（本リリースで"読書する曜日/頻度"を反映）

#### 5. 読了

読了にして一覧がスッキリ（達成感）。

- 読了はユーザーが手動で切替
- 現在ページが読了対象ページ数に到達した場合は、詳細画面に「読了にする」ボタンを強調表示

### 本リリース

1. 買ったばかりの技術書を登録し「2週間後」を賞味期限に設定
2. 読書可能な時間帯（例：平日19:00-21:00）を登録
3. Googleカレンダー連携 → 指定時間帯に30分の読書枠が自動挿入される
4. 期限3日前、メールで「あと3日です」と通知。書影もセピア調に変化  
   - **「3日前」は（期限日 - 3日）のJST 00:00基準で判定し、同一ユーザー・同一本につき1回のみ発火**
5. 積読タイムラインで他のユーザーの進捗を見てモチベーションアップ
6. 読了時にシェア用ページを生成（OGP対応）

## サービスの差別化ポイント・推しポイント

### 差別化ポイント

#### 1. 「記録」ではなく「予定化」まで面倒を見る

既存の読書管理サービス（ブクログ等）は、読んだ本・読みたい本の**ログ**が中心で、「読む行動」はユーザー任せになりがちです。

Yomikiriは、登録した本を**読書タスクとして予定化**し、「読む時間」を先に確保するところまで支援します。

| フェーズ | 内容 |
|----------|------|
| MVP | ワンクリックで「Googleカレンダーの予定作成画面（テンプレ入力済み）」へ遷移 |
| 本リリース | ユーザーが登録した読書可能時間帯に、30分枠の読書予定を自動挿入 |
| 将来 | freebusy APIで空き時間を解析し、最適な時間に自動挿入 |

> **Note**  
> - アプリ内にカレンダーUIは持たず、普段使っているGoogleカレンダーに予定を追加してもらう設計  
> - MVPは都度追加方式（繰り返し予定はGoogle側で設定可能）  
> - MVPはGoogle側の変更/削除を追跡しない（まず予定化の効果を検証）  

#### 2. 「期限」を"行動に変換できる形"で提示する

ただ締切日を見せるだけでなく、期限から逆算して**今日やるべき量**まで落とし込みます。

- 残ページ数・残日数から、**1日あたりの必要読書ページ数（ノルマ）**を自動計算
- ノルマは以下のタイミングで更新
  - 進捗更新（現在ページの更新）時に再計算
  - 日付が変わったタイミング（例：毎日深夜の定期処理）でも再計算
- → 「何をどれだけ読めばいいか」が常に明確に

| フェーズ | 追加機能 |
|----------|----------|
| 本リリース | 「読書する曜日/頻度」設定（例：月・水・金のみ、または週3日）を追加し、"読む日"のみで逆算 |
| 本リリース | 読書スピード計測を導入し、時間（予定枠）⇄ページ（ノルマ）の換算精度を向上 |

#### 3. 期限の緊急度を"体感できる"UI（賞味期限ビジュアライザー）

期限が近づくほど書影が色褪せる（セピア＋透明度）ことで、**一覧を見ただけで「早く読まないとまずい本」**が分かります。

数字の締切を"体感"に変えて、先延ばしを防ぎます。

| 残り日数 | 変化 |
|----------|------|
| 7日以下 | セピア/透明度：弱 |
| 3日以下 | セピア/透明度：中 |
| 1日以下 | セピア/透明度：強 |

> **Note**  
> MVP時点では固定値、将来的にユーザー設定可能にする可能性あり

#### 4. 相互の刺激で継続を支援する（積読タイムライン）

本リリースでは、共有設定した本のみがタイムラインに表示され、他のユーザーの進捗や「期限への焦り」が可視化されます。

一人では挫折しがちな読書を、**相互の刺激**によって習慣化させます。

| 項目 | 内容 |
|------|------|
| デフォルト設定 | 非公開。本ごとに共有を明示的にONにした場合のみ表示 |
| 公開される情報 | 本のタイトル・期限・進捗（%）のみ（メモ等は非公開） |
| 表示名 | **ユーザー名はニックネーム表示（実名不要）** |
| 進捗の計算 | 進捗(%) = 現在ページ / 読了対象ページ数 × 100 |
| 表示形式 | イベント単位（新規登録時 / **期限3日前** / 読了時） |
| 期限3日前の定義 | **（期限日 - 3日）のJST 00:00基準で判定し、同一ユーザー・同一本につき1回のみ発火** |

### 推しポイント

| ポイント | 内容 |
|----------|------|
| 推し1：意志に頼らず、予定化で「読む時間」を先に押さえる | 「いつ読もうか」を考える認知負荷が減り、忙しくても後回しになりにくい。MVPでも予定作成までの手間を最小化（テンプレ入力済み）、本リリースでは自動挿入でさらに強制力を高める |
| 推し2：「今日は何ページ？」が出るから、着手が軽い | 期限と進捗からノルマが出ることで、「時間がないから無理」を「今日は◯ページ読めばOK」に変換。迷いが減り、日々の読書が"タスク化"される |
| 推し3：期限が近いほど焦る設計で、先延ばしを抑える | 賞味期限ビジュアライザーにより、期限が迫るほど視覚的に緊急度が上がる。締切を「ただの数字」にしないことで、行動を引き出す |

| 積読タイムライン機能 | 他のユーザーの積読状況や進捗を閲覧可能。共有設定した本のみ表示（デフォルト非公開）。イベント単位で表示（新規登録時／**期限3日前**／読了時） |
| 読書スピード計測 | タイマー機能で1ページあたりの読書時間を計測・保存し、ノルマ計算の精度を向上 |
| バックグラウンド処理 | GitHub Actions（cron）でrake taskを定期実行（毎日の期限チェック・通知送信）。失敗時はGitHub Actionsの通知で検知し、再実行できる運用を想定 |

> 延長は無制限に可能ですが、延長回数を目立つ位置に表示します。  
> 本リリースでは延長もイベント化（例：「期限を延長しました」）することで、先延ばしコストを可視化する予定です。

## 使用する技術スタック

### 実装イメージ（CRUD以外）

#### Googleカレンダー連携

| フェーズ | 実装方法 |
|----------|----------|
| MVP（簡易版） | Google Calendar の event create URL を生成して遷移（OAuth不要）。**予定タイトル / 説明文 / 所要時間（終了時刻）**を事前入力し、日時はユーザーが選択 |
| 本リリース（自動版） | OmniAuth + google-api-client でOAuth認証し、ユーザー指定時間帯に events.insert |
| 将来（空き時間解析版） | freebusy API で空き時間を取得し、最適な時間に自動挿入 |

#### ノルマ計算ロジック

残りページ数を $P$（読了対象ページ数 - 現在ページ）、残り日数を $D$（**今日を含む残日数。期限当日は $D=1$**）とした時、1日あたりの必要ページ数 $N$ を **切り上げ** で算出します。

$$N = \lceil P / D \rceil$$

更新タイミング：
- 進捗更新時（現在ページ更新時）
- 日付切り替わり（毎日深夜の定期処理）

#### バックグラウンド処理

GitHub Actions（cron）でrake taskを実行し、毎日決まった時間に以下を実行します。

- ノルマ再計算
- 期限イベント生成（例：期限3日前、読了など）
- 通知対象抽出（本リリースで送信）

- APIキー等はGitHub Secretsで管理
- 定期処理はJST基準で実行（GitHub ActionsのcronはUTCのため、実行時刻をJSTに換算して設定）
